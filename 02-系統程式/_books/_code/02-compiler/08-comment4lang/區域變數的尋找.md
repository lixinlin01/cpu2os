## ccc: 上述 C 語言程式中，遞迴函數呼叫時，是如何取得正確的區域變數的呢？




在上述的 C 語言程式碼中，能夠完美支援遞迴呼叫且不會讓區域變數互相干擾，核心的魔法在於實作了**「呼叫堆疊 (Call Stack)」**與**「堆疊幀 (Stack Frame)」**的機制。

具體來說，是透過全域的 `Frame stack[1000];` 陣列和一個堆疊指標 `int sp = 0;` 來控制的。

我們可以拆解成以下三個關鍵步驟來理解：

### 1. 資料結構的隔離 (Stack Frame)
在 C 語言的設計中，定義了 `Frame` 這個結構體：
```c
typedef struct {
    char names[100][32];  // 區域變數名稱
    int values[100];      // 區域變數值
    int count;            // 目前幀的變數數量
    // ...略（用來儲存返回位址與參數的欄位）
} Frame;

Frame stack[1000]; // 虛擬機的記憶體空間，包含多個幀
int sp = 0;        // Stack Pointer，永遠指向「當下正在執行的那一層」
```
每個 `Frame` 就像一個**「獨立的房間」**，裡面有專屬的記事本（`names` 和 `values` 陣列）。每次函數執行，都只能在自己的房間裡寫筆記。

### 2. 當發生遞迴呼叫時 (`CALL` 指令)
當虛擬機讀到 `CALL` 指令（例如在 `fact(3)` 裡面又呼叫了 `fact(2)`），程式會執行以下動作：

```c
sp++;                // 1. 關鍵！將堆疊指標往上推，進入下一個「新房間」
stack[sp].count = 0; // 2. 清空這個新房間的變數數量，代表這是一個全新的作用域
// ...傳遞參數與跳轉 PC 略...
```
**這就是遞迴能運作的核心**。雖然呼叫的是同一個函數，但因為 `sp` 增加了（例如從 `sp = 1` 變成 `sp = 2`），虛擬機在陣列中開啟了一個**全新的 `Frame` 結構**。舊的變數狀態（`sp = 1`）被完整保留在陣列的上一層，毫髮無傷。

### 3. 變數的讀取與寫入 (`get_var` 與 `set_var`)
為什麼在 `fact(2)` 裡面存取變數 `n`，不會去改到 `fact(3)` 裡面的 `n` 呢？
我們來看看變數操作的輔助函式：

```c
int get_var(const char *name) {
    // ...略
    for (int i = 0; i < stack[sp].count; i++) 
        if (strcmp(stack[sp].names[i], name) == 0) return stack[sp].values[i];
}

void set_var(const char *name, int val) {
    for (int i = 0; i < stack[sp].count; i++) 
        if (strcmp(stack[sp].names[i], name) == 0) { stack[sp].values[i] = val; return; }
    // 新增變數
    strcpy(stack[sp].names[stack[sp].count], name);
    stack[sp].values[stack[sp].count++] = val;
}
```
有沒有發現？這兩個函式**死死地綁定了 `stack[sp]`**！
它們在尋找變數 `name` (例如 `"n"`) 時，**絕對不會跨層**去找。它只會在目前 `sp` 指向的那個 `Frame` 裡面掃描。因此，每一層遞迴裡面的 `n` 都是在各自的 `names/values` 陣列中獨立建檔的。

### 4. 遞迴返回時 (`RET_VAL` 指令)
當最深層的遞迴執行完畢遇到 `return` 時：

```c
int ret_val = get_var(q.arg1);          // 1. 先在當前層(房間)把計算結果拿出來
// ...略...
sp--;                                   // 2. 關鍵！銷毀當前的堆疊幀，退回上一層房間
set_var(target_var, ret_val);           // 3. 把回傳的結果寫入「上一層(Caller)」的變數中
```
執行 `sp--` 後，虛擬機的「視角」就回到了上一層遞迴。因為 `sp` 減回去了，接下來所有的 `get_var` 和 `set_var` 又會重新對準上一層的變數陣列。剛剛深層遞迴所用的 `Frame` 就這樣被邏輯上「銷毀（遺棄）」了。

---

### 總結（圖解概念）
假設我們執行遞迴 `fact(3)` 呼叫 `fact(2)` 呼叫 `fact(1)`，虛擬機記憶體的狀態變化如下：

1. **進入 fact(3)** : `sp = 1`, `stack[1]` 裡記錄變數 `n = 3`
2. **進入 fact(2)** : `sp = 2`, `stack[2]` 裡記錄變數 `n = 2`
3. **進入 fact(1)** : `sp = 3`, `stack[3]` 裡記錄變數 `n = 1`
   👉 *此時虛擬機要讀取 `n`，因為 `sp=3`，它只看 `stack[3]`，所以得到 1。*
4. **fact(1) 返回** : 取得結果，並執行 `sp--` 變成 `sp = 2`。
   👉 *控制權回到 fact(2)，此時虛擬機若去讀 `n`，因為 `sp=2`，它看到的就是 `stack[2]` 裡面的 2，資料完全沒被污染！*

這套 `sp++` (進入) 與 `sp--` (離開) 的陣列切換設計，正是所有現代作業系統與虛擬機處理函數呼叫與區域變數的基礎原理。

## ccc: 但是 global 變數的尋找機制，和區域變數不同嗎？




這是一個非常敏銳且專業的問題！你完全點出了這份「迷你編譯器」在設計上的一個**刻意簡化（或稱為盲點）**。

在我們剛才討論的 C 語言（以及改寫後的 Rust）程式碼中，**全域變數和區域變數的尋找機制其實是「完全一樣」的，這導致了一個結果：函數內部其實「看不到」全域變數！**

讓我們來拆解為什麼會這樣，以及在「真正的編譯器」中應該怎麼處理。

### 1. 目前程式碼的現況：函數看不到全域變數

回頭看 C 語言中的 `get_var`：
```c
int get_var(const char *name) {
    if (isdigit(name[0])) return atoi(name);
    // 注意這裡！它【只】尋找 stack[sp]
    for (int i = 0; i < stack[sp].count; i++) 
        if (strcmp(stack[sp].names[i], name) == 0) return stack[sp].values[i];
    return 0; // 找不到就回傳 0
}
```

* 當程式在**全域範圍**執行時（還沒呼叫任何函數），`sp = 0`。此時讀寫變數都在 `stack[0]`，這就是全域變數。
* 當程式**進入函數**時（例如 `CALL`），`sp` 變成了 1、2、3...。此時 `get_var` 和 `set_var` **被死死地限制在 `stack[sp]` 裡面**。

**這會發生什麼事？**
假設你在全域定義了 `G = 100`，然後進入函數 `func()`：
* 如果你在 `func()` 裡面讀取 `G`：虛擬機在 `stack[sp]` 裡找不到 `G`，它**不會**退回去 `stack[0]` 找，而是直接回傳 `0`。
* 如果你在 `func()` 裡面寫入 `G = 50`：虛擬機會在目前的 `stack[sp]` 裡面「新建」一個名為 `G` 的區域變數。真正的全域變數 `G` 完全沒被改到。

這在編譯器設計中稱為**「純局部作用域 (Purely Local Scope)」**，通常只出現在極度簡化的教學模型中。

---

### 2. 真正的編譯器如何處理？（作用域鏈 Scope Chain）

如果要讓這個編譯器支援「函數讀取全域變數」，我們必須修改變數的尋找機制，引入**作用域鏈 (Scope Chain)** 的概念。

尋找變數的正確邏輯應該是：**「先找自己的房間（區域變數），如果找不到，再去大廳（全域變數）找。」**

如果我們要修復這個機制，可以這樣改寫 C 語言的虛擬機：

#### 修正 `get_var` (讀取變數)
```c
int get_var(const char *name) {
    if (isdigit(name[0])) return atoi(name);
    
    // 1. 先在當前的區域作用域 (stack[sp]) 尋找
    for (int i = 0; i < stack[sp].count; i++) {
        if (strcmp(stack[sp].names[i], name) == 0) return stack[sp].values[i];
    }
    
    // 2. 如果目前在函數內 (sp > 0)，且區域找不到，就去全域 (stack[0]) 找
    if (sp > 0) {
        for (int i = 0; i < stack[0].count; i++) {
            if (strcmp(stack[0].names[i], name) == 0) return stack[0].values[i];
        }
    }
    
    return 0; // 真的都找不到才回傳 0
}
```

#### 修正 `set_var` (寫入變數)
寫入變數的邏輯比較複雜，因為這牽涉到語言的「語意設計」。當你寫下 `X = 10` 時，你是要修改全域的 `X`，還是要在區域新建一個 `X`？

一般類似 C/Python 的直覺設計是：**「如果全域已經有這個變數，就覆寫全域；如果全域沒有，就在區域新建。」**
*(註：Python 其實需要 `global` 關鍵字，但為了簡單說明，我們採取最單純的覆蓋策略)*

```c
void set_var(const char *name, int val) {
    // 1. 先檢查區域是否已經有這個變數，有就更新
    for (int i = 0; i < stack[sp].count; i++) {
        if (strcmp(stack[sp].names[i], name) == 0) { 
            stack[sp].values[i] = val; 
            return; 
        }
    }
    
    // 2. 如果區域沒有，去檢查全域是不是有這個變數，有就更新全域
    if (sp > 0) {
        for (int i = 0; i < stack[0].count; i++) {
            if (strcmp(stack[0].names[i], name) == 0) { 
                stack[0].values[i] = val; 
                return; 
            }
        }
    }
    
    // 3. 區域跟全域都沒有？那就在當前區域 (stack[sp]) 新建一個變數！
    strcpy(stack[sp].names[stack[sp].count], name);
    stack[sp].values[stack[sp].count++] = val;
}
```

### 3. 進階探討：閉包 (Closure) 與靜態連結 (Static Link)
你提到的問題，觸及了編譯器後端非常重要的一環——**環境解析 (Environment Resolution)**。

1. **兩層結構 (C 語言風格)**：就像上面的修正，只有「區域 (sp)」和「全域 (0)」。
2. **多層結構 (JavaScript/Python 風格)**：如果你可以在函數裡面定義函數（巢狀函數），那找變數的過程就會變成 `sp` -> `sp的母函數` -> `sp的祖母函數` -> ... -> `全域`。
   * 在這種情況下，編譯器會在 Stack Frame 裡面多加一個指標，稱為 **Static Link (靜態連結)** 或 **Environment Pointer**，它會指向「定義這個函數時的外層環境」，藉此一層一層往外找。這也就是 **閉包 (Closure)** 能夠記住外部變數的底層原理。

總結來說，你看到的這份玩具程式碼為了極度精簡，犧牲了全域變數的穿透性（變成了純區域作用域）。如果要支援，就是要從 `get_var` 下手，讓它具備「往外層 `stack[0]` 探查」的能力！
